# Сервис мониторинга репостов ВК по сеткам ЮВАО

## 1. Цель и контекст

Внутренний веб-сервис для сотрудников управ районов ЮВАО (Капотня, Выхино-Жулебино, Марьино, Кузьминки, Люблино). Позволяет:

1. Ввести ссылку на пост-первоисточник во ВКонтакте.
2. Выбрать сетку (один из пяти районов).
3. Получить отчёт о том, какие ресурсы из сетки сделали репост этого поста, с заполненной таблицей по форме «Распространение на медиаресурсах АО».
4. Скачать отчёт в Excel.

Окно поиска репостов — 8 дней с даты публикации первоисточника. В отчёт попадают только ресурсы, где репост найден.

## 2. Технологический стек

- **Бэкенд**: Python 3.11+, FastAPI, SQLAlchemy 2.x, Alembic (миграции).
- **БД**: PostgreSQL 15+.
- **Фронт**: серверный рендеринг через Jinja2 + HTMX для интерактивных кусков (без SPA, чтобы не множить сущности — это внутренний инструмент с ограниченным числом экранов).
- **Стили**: Tailwind CSS через CDN или Pico.css — минимализм, не нужен build-step.
- **ВК API**: прямые вызовы через `httpx` (асинхронный клиент). Библиотека `vk_api` не нужна — методов мало.
- **Excel**: `openpyxl`.
- **Фоновые задачи**: для MVP — `BackgroundTasks` из FastAPI. Если задач станет много, мигрировать на Celery + Redis.
- **Аутентификация**: сессионные куки, пароли хешируются bcrypt (`passlib`).
- **Деплой**: Docker + Docker Compose (приложение + Postgres).

## 5. Интеграция с ВК API

### 5.1 Получение токена
Сервис использует **сервисный ключ доступа** одного ВК-приложения (тип «Standalone»). Создаётся вручную в `https://dev.vk.com` владельцем сервиса, хранится в env как `VK_SERVICE_TOKEN`. Сотрудники управ ВК-токенов не имеют — все запросы идут от приложения.

В каждом запросе параметры:
- `access_token=<VK_SERVICE_TOKEN>`
- `v=5.199` (зафиксировать в `config.py`)
- `lang=ru`

Базовый URL: `https://api.vk.com/method/<method_name>`.

### 5.2 Используемые методы

| Метод | Когда вызываем | Что берём |
|---|---|---|
| `utils.resolveScreenName` | при добавлении ресурса в админке | вернёт `object_id` и `type` (`user`/`group`) по screen_name |
| `wall.getById` | при создании отчёта, чтобы достать дату первоисточника | `date` поста |
| `wall.get` | при сборе отчёта, по каждому ресурсу сетки | посты со стены, фильтруем по `copy_history` |
| `groups.getById` | при сборе отчёта для ресурсов типа группа | `name`, `members_count` |
| `users.get` | при сборе отчёта для личных страниц | `first_name`, `last_name`, `followers_count` |

### 5.3 Парсинг URL первоисточника

Принимаем форматы:
- `https://vk.com/wall-12345_67890`
- `https://vk.com/wall12345_67890`
- `https://vk.com/somegroup?w=wall-12345_67890`
- `https://m.vk.com/wall-12345_67890`

Регулярка: `r'wall(-?\d+)_(\d+)'` — первая группа `owner_id` (может быть отрицательной), вторая `post_id`. Если регулярка не сматчилась — отдать пользователю ошибку «не похоже на ссылку на пост ВК».

### 5.4 Алгоритм поиска репостов

Псевдокод ядра в `services/repost_finder.py`:

```
def find_reposts(report):
    source_date = vk.wall.getById(report.source_owner_id, report.source_post_id).date
    window_end = source_date + 8 days
    
    resources = db.resources.filter(network_id=report.network_id, is_active=True)
    
    for resource in resources:
        # Получаем последние посты (хватит 100 для окна в 8 дней у любого нормального ресурса)
        posts = vk.wall.get(owner_id=resource.vk_owner_id, count=100)
        
        for post in posts:
            if post.date < source_date:
                break  # дальше только старее
            if post.date > window_end:
                continue
            
            if 'copy_history' not in post or not post.copy_history:
                continue
            
            original = post.copy_history[0]
            if (original.owner_id == report.source_owner_id 
                    and original.id == report.source_post_id):
                # Нашли репост
                save_report_row(
                    resource=resource,
                    repost_owner_id=post.owner_id,
                    repost_post_id=post.id,
                    repost_date=post.date,
                    views_count=post.views.count if 'views' in post else None,
                    followers_count=fetch_followers(resource),
                )
                break  # один ресурс — один репост
        
        sleep(0.34)  # лимит 3 rps, страхуемся
```

`fetch_followers(resource)`:
- если `vk_owner_id < 0` → `groups.getById(group_id=abs(owner_id), fields='members_count')` → `members_count`
- иначе → `users.get(user_ids=owner_id, fields='followers_count')` → `followers_count`

Названия ресурсов (для колонки «Наименование канала/сообщества») берём из тех же ответов и кладём в `report_rows` или джойним при экспорте — на выбор.

### 5.5 Оптимизация: метод `execute`

VK API позволяет упаковать до 25 вызовов в один HTTP-запрос через метод `execute` со скриптом на VKScript. Для MVP можно обойтись последовательными вызовами (50 ресурсов × ~0.34с = 17 секунд, терпимо). Если будет медленно — переделать обход на `execute`, скрипт пишется на ВКшном диалекте JavaScript, пакует `wall.get` для 25 ресурсов разом.

### 5.6 Обработка ошибок ВК API

Ответ ВК всегда вида `{"response": ...}` или `{"error": {"error_code": N, "error_msg": "..."}}`. Обязательно обрабатывать:

- `error_code=5` — токен невалиден → залогировать критичную ошибку, упасть с понятным сообщением админу.
- `error_code=6` — too many requests → backoff и ретрай.
- `error_code=14` — капча → залогировать `captcha_sid` и `captcha_img`, пометить отчёт `failed` с понятным сообщением «ВК запросил капчу, обратитесь к администратору». Капча на сервисном токене при чтении публичных стен бывает редко.
- `error_code=15` или `30` — доступ запрещён (профиль закрыт, группа удалена) → пометить ресурс в логе отчёта как «недоступен», продолжить с остальными.
- Сетевые ошибки → 3 ретрая с экспоненциальным бэкоффом.

## 6. Логика формирования отчёта

1. Пользователь на странице «Новый отчёт» вводит:
   - Ссылку на первоисточник.
   - Выбирает сетку.
   - Вписывает наименование инфоповода (свободный текст).
2. Бэкенд парсит URL, валидирует. Создаёт запись `reports` со статусом `pending`.
3. Запускает `BackgroundTasks` → `find_reposts(report_id)`.
4. Статус меняется на `running`, по завершении — `done` или `failed`.
5. Фронт через HTMX опрашивает `/reports/{id}/status` каждые 2 секунды, после `done` редиректит на страницу отчёта.

## 9. Аутентификация

Самый простой вариант, достаточный для внутреннего инструмента:

- Таблица `users` с bcrypt-хешами.
- Логин/пароль → сессионный токен в подписанной куке (FastAPI `itsdangerous` или встроенный `SessionMiddleware`).
- Middleware проверяет наличие сессии на всех маршрутах кроме `/login`.
- Роль `admin` нужна для доступа к `/admin/*`.
- Первый администратор создаётся скриптом-сидером (`scripts/create_admin.py`), читает логин и пароль из переменных окружения при первом запуске.

## 13. Подводные камни и edge cases

1. **Закрытый профиль ЛОМа**. `wall.get` вернёт `error_code=30`. Логировать и продолжать с остальными ресурсами. В UI отчёта — раздел «Ресурсы, недоступные при сборе» (опционально, не обязательно для MVP).

2. **Удалённая группа / удалённый аккаунт**. Аналогично: в логе и в UI отметить, в отчёт не попадает.

3. **Несколько репостов одного и того же первоисточника на одной странице**. По логике задачи — берём первый найденный (`break` после первого матча).

4. **Репост репоста**. Если ресурс репостнул не сам первоисточник, а чей-то репост — формально это будет другая `copy_history`. ВК в `copy_history` хранит цепочку, и `copy_history[-1]` — самый исходный пост. Решение: при матчинге проверять не только `copy_history[0]`, а всю цепочку — если в любом элементе совпали owner_id и post_id, считаем репостом. Это корректнее для отчётности.

5. **Источник из закрытого паблика / приватного профиля**. `wall.getById` упадёт, отчёт сразу `failed` с понятным сообщением.

6. **Просмотры отсутствуют**. У старых постов или некоторых форматов поля `views` нет — оставлять `null`, в Excel пустую ячейку.
